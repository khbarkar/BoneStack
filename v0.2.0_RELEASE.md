## 🎉 What's New in v0.2.0

BoneStack v0.2.0 introduces **comprehensive layer analysis capabilities**, enabling forensic inspection of Docker image layers, bloat detection, and size optimization analysis through an intuitive interactive UI.

### Major Additions

#### 1. 📊 Layer Analysis Engine (740 lines)
Complete backend infrastructure for analyzing Docker image layers:

- **Layer Extraction**: Parse Docker image history to extract layer metadata
  - Layer ID (SHA256 digest)
  - Size in bytes
  - Creation timestamp
  - Dockerfile instruction (command)
  
- **Bloat Detection**: Recognize and flag optimization opportunities
  - 8+ predefined bloat patterns
  - Package manager cache detection (apt, yum, apk, npm, pip, gem)
  - Build tools and development metadata
  - Estimated savings for each pattern
  - Confidence scoring (0.0-1.0)

- **Layer Comparison**: Generate diffs between consecutive layers
  - Track files added/modified/removed
  - Size change analysis
  - Build impact assessment

- **Recommendations Engine**: Suggest specific optimizations
  - Cleanup commands (e.g., `apt-get clean`)
  - Multi-stage build suggestions
  - Dockerfile optimization advice

## 🚀 Usage Guide

### Starting BoneStack
```bash
./bonestack
```

### Workflow to Analyze Layers

1. **Main Menu**
   - Navigate: ↑/↓ or j/k
   - Select: Enter

2. **View Images**
   - Select an image and press Enter

3. **Image Details**
   - Press **l** to view layers

4. **Layer Inspection**
   ```
   ↑/↓ or j/k  → Navigate layers
   Enter        → View layer details & bloat
   s            → View size breakdown
   b            → Return to image details
   ```

5. **Layer Details**
   - See bloat patterns
   - Get removal recommendations
   - View confidence score

6. **Size Breakdown**
   - Visual comparison of layer sizes
   - Cumulative growth tracking
   - Identify largest contributors

### Keyboard Shortcuts

| Key | Action |
|-----|--------|
| ↑ / k | Navigate up |
| ↓ / j | Navigate down |
| Enter | Select / View details |
| l | View layers (from image details) |
| s | Size breakdown (from layer list) |
| b | Go back |
| q | Quit |

## 🛠️ Technical Architecture

### Modular Design
```
internal/layers/
├── types.go       # Data structures
├── analyzer.go    # Layer analysis engine
├── diff.go        # Diff generation
├── bloat.go       # Bloat detection
└── layers_test.go # Unit tests

internal/ui/app.go       # Updated with 3 new screens
internal/models/state.go # Extended AppState
```

### Separation of Concerns
- **Layer Package**: Independent of UI
  - Can be used in CLI tools
  - Can be used in APIs
  - Can be used in batch processing

- **UI Package**: Clean abstraction
  - Uses layer package interfaces
  - Handles all screen rendering
  - Manages user input

### Data Flow
```
Docker Image → GetImageHistory() → ImageLayers
                                       ↓
                           [LayerAnalyzer]
                                       ↓
                           LayerAnalysis[]
                                       ↓
                       [BloatDetector] → Recommendations
                                       ↓
                           [DiffEngine] → LayerDiff[]
                                       ↓
                           [UI Screens] → Display
```

## 📊 Bloat Detection Algorithm

### Pattern Matching
1. Parse Dockerfile command for layer
2. Check against known patterns
3. Match keywords in command
4. Return bloat items if matched

### Estimation
- **File Count**: ~5KB average per file
- **Size Estimation**: Pattern database values
- **Confidence**: 0.85 (default, improved with tar inspection in v2.5)

### Example
```go
Command: "RUN apt-get install -y curl wget"
↓
Match: apt_cache pattern
↓
Bloat Item:
  Pattern: "apt_cache"
  Description: "apt cache not cleaned"
  EstimatedSize: 80MB
  Removable: true
```

## 🐛 Known Limitations

1. **File Count Estimation**: Uses heuristic (~5KB/file)
   - Real implementation in v2.5 with tar inspection

2. **Bloat Detection**: Pattern-based on Dockerfile commands
   - Requires actually executing or inspecting tar for accuracy

3. **Size Breakdown**: Heuristic file counts
   - Real data in v2.5

4. **Layer Metadata**: Requires Docker daemon access
   - Limitation of Docker API
